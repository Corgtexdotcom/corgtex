# E2E Agent Auth Resolution Plan

## Background
The previous testing found that the `AGENT_API_KEY` was correctly authenticating in local environments but failing (`401 Missing Session`) against Railway production. 

## Completed Fixes

### 1. Railway Environment Configuration
- Injected `AGENT_API_KEY` into the production deployment via the CLI.
- Verified that Railway's Next.js container rebuilt and correctly loaded the new environment variable.

### 2. Codebase Agent Routing Bugs
After the env var was properly set, comprehensive E2E testing using `sandbox/e2e-test.mjs` revealed 4 endpoints that were improperly rejecting or crashing on Agent actors:

1. **`GET /api/workspaces`**: Threw `400 "Only users can list workspaces"`. **Fix**: Switched from `listWorkspaces` to `listActorWorkspaces` which properly supports agents.
2. **`GET /api/workspaces/[wsId]/check-ins`**: Threw `401 Unauthorized` for agents because `requireWorkspaceMembership` legitimately returns `null` for non-human actors. **Fix**: Explicitly mapped `null` memberships to an empty array for agents, as check-ins are strictly human-relative.
3. **`POST /api/workspaces/[wsId]/actions` & `circles`**: Threw `403` because passing an explicit `null` membership bypasses the normal agent authorization fallback. **Fix**: Coerced `null` membership to `undefined` centrally in `apps/web/lib/route-handler.ts` so that domain layer authorization handles agent actors naturally.

## Walkthrough & Verification

I have developed a full automation suite `sandbox/e2e-test.mjs` which locally validates everything working at `app.corgtex.com` (which was already deployed from `main`).
Merging this PR will deploy these same fixes to `corgtexweb-production.up.railway.app` or any other target using this codebase.
